# HTML 내 'spec'/'plan' 한글화 계획 (초안 — 소스 미반영)

작성일: 2026-07-16
대상: `기관/total/서울시_5개구청_사업제안기획안.html` 및 이를 생성하는 `build_html_report.py`,
그리고 그 원문 소스인 `기관/{구청}/spec/*.txt`, `기관/{구청}/plan/*.txt`, `기관/{구청}/bank_ideas_draft.txt`

## 1. 현황 조사 결과

HTML 안에서 `spec`은 654회, `plan`은 106회 등장한다. 전수 확인 결과 용법은 두 가지로 나뉜다.

### 용법 A. 파일 코드 참조 (다수, 예: `spec/07`, `plan/02`, `spec/01의`)
- 각 구청 조사자료 폴더의 파일 번호를 가리키는 출처 표기.
  예: `spec/07 확인율27% 미확인`, `plan/02 IT-1 "노원구 통합 사업추진현황..."`
- 이 표기는 실제로 `기관/{구청}/spec/07_홈페이지검색확인결과.txt`,
  `기관/{구청}/plan/02_IT디지털기획_사업제안.txt` 같은 **실제 파일명과 대응**되어 있어,
  검증 근거를 추적할 때 "몇 번 파일의 어떤 내용인지" 되짚어 볼 수 있게 하는 역할을 한다.
- 이 표기가 5개 구청의 `plan/05_검증결과.txt` 27곳, `plan/00~04` 파일들에도 광범위하게
  박혀 있고, `bank_ideas_draft.txt`에도 `spec/02(1장-4)`처럼 세부 인덱스까지 붙어 인용된다.

### 용법 B. 조사가 붙은 일반 명사형 (소수, 예: `spec에서`, `plan이`, `spec의`)
- "spec 조사에서 확인된 문제", "plan이 이미 설계한", "spec의 패턴분석" 등
- 문장 안에서 조사(-에서/-이/-의/-은)가 그대로 로마자 뒤에 붙는 형태.

### 텍스트가 실제로 존재하는 3개 층
1. **원본 조사/제안 텍스트 파일** (`기관/{구청}/spec/00~09.txt`, `plan/00~05.txt`,
   `bank_ideas_draft.txt`) — spec/plan 표기가 가장 많이 박혀 있는 원천. 도봉구만 봐도
   plan 폴더 6개 파일에서 spec/plan 언급이 57건.
2. **`build_html_report.py`의 하드코딩 문자열** — `key_problems`, `verdicts` 딕셔너리
   안에 "spec 조사에서 실제로 확인된 문제(검색 미확인 27%...)" 같은 요약 문장이
   파이썬 코드 안에 직접 박혀 있음 (구청별 5개 verdict + 20개 problem 항목).
3. **HTML 정적 스캐폴딩 텍스트** — footer의 "근거자료: ... plan 제안서 타당성 검증 결과 ·
   은행 결합 아이디어 초안(bank_ideas_draft)", 서브탭 배지 등.

## 2. 번역 방향 제안

'spec'과 'plan'은 이미 이 프로젝트에서 굳어진 약어이며, 각각:
- **spec** = "조사자료" / "조사보고서" (구정 현황·예산·홈페이지 확인·민원게시판 조사)
- **plan** = "제안서" / "사업제안" (IT/금전지원 제안 및 검증)

을 가리킨다. 두 용법을 구분해 다음과 같이 치환하는 방향을 제안한다.

### 2-1. 파일 코드 참조 (`spec/07`, `plan/02` 등)
숫자 인덱스는 실제 파일 추적성을 위해 유지하되, 접두어만 한글로 바꾸는 안:

| 현재 | 제안 |
|---|---|
| `spec/07` | `조사자료07` 또는 `조사07` |
| `plan/02` | `제안서02` 또는 `제안02` |
| `spec/07의` | `조사자료07의` |
| `plan/00~04` | `제안서00~04` |

**검토 포인트**: 붙임표/구분자 없이 "조사자료07"처럼 붙이면 가독성이 다소 떨어질 수 있어
`조사자료 07` (공백 추가) 또는 `조사자료#07`(해시 구분) 형태도 후보로 고려.
파일 시스템상 실제 파일명이 `07_홈페이지검색확인결과.txt`이므로, 치환 시 아예
"조사자료 07(홈페이지 검색확인)"처럼 파일 설명을 붙이는 안도 가능하나 문장이 길어지는
단점이 있어 **숫자 인덱스만 유지하는 단순 치환을 기본안으로 제안**.

### 2-2. 조사 결합형 (`spec에서`, `plan이`, `spec의` 등)
문맥상 "조사자료" 또는 "조사보고서"로 치환:

| 현재 | 제안 |
|---|---|
| `spec 조사에서 확인된` | `조사자료에서 확인된` (※ "spec 조사"는 이미 "조사"가 중복 표기되어 있어, 치환 시 "조사자료에서"로 정리하면 중복 해소) |
| `plan이 이미 설계한` | `제안서가 이미 설계한` |
| `spec의 패턴분석` | `조사자료의 패턴분석` |
| `plan은 spec 사업목록에 없는` | `제안서는 조사자료 사업목록에 없는` |

### 2-3. 복합 표현 (`spec/plan`, `spec·plan`)
- `spec/plan 전체 대조` → `조사자료·제안서 전체 대조`
- 소개 문구에서 "spec(조사)·plan(제안서 타당성 검증 결과)" 처럼 최초 1회는 병기 방식도 검토

### 2-4. footer 및 배지 등 고정 문구
| 위치 | 현재 | 제안 |
|---|---|---|
| footer | `plan 제안서 타당성 검증 결과` | `제안서 타당성 검증 결과` (plan 삭제, 이미 "제안서"라는 한글 병기 있어 중복 제거) |
| footer | `은행 결합 아이디어 초안(bank_ideas_draft)` | `은행 결합 아이디어 초안(bank_ideas_draft.txt)` 유지 검토 — 이건 실제 파일명이라 영문 유지가 안전 (파일명은 번역 대상에서 제외 권장) |

## 3. 변경이 필요한 파일 목록과 손댈 범위

### (1) 원본 텍스트 파일 — 총 5구청 × (spec 9~10개 + plan 6개 + bank_ideas_draft 1개) = 약 80개 파일
- `기관/{도봉,노원,광진,동대문,동작}/spec/*.txt` 전체
- `기관/{도봉,노원,광진,동대문,동작}/plan/*.txt` 전체
- `기관/{도봉,노원,광진,동대문,동작}/bank_ideas_draft.txt`
- **영향도**: 가장 크다. 이 파일들이 "1차 소스"이므로 여기를 고치면 HTML 재생성 시
  자동으로 반영된다. 단, 80개 파일에 걸쳐 "spec/NN", "plan/NN" 수백 건을 정규식으로
  일괄 치환해야 하므로 스크립트화가 필요(수작업 금지 수준의 반복 작업).
- **리스크**: 파일 안에 있는 "spec/02(1장-4)"처럼 세부 하위 인덱스(`1장-4`, `2-36` 등)
  표기와 결합된 경우 정규식이 과도하게 넓거나 좁게 걸릴 수 있어 치환 전/후 diff 검토 필요.

### (2) `build_html_report.py` — 1개 파일
- `key_problems` 딕셔너리(5개 구청 × 4개 항목 = 20개 문장) 중 "spec/07", "spec/08" 등
  파일 코드가 하드코딩된 부분
- `verdicts` 딕셔너리(5개 구청 × 1개 문장) 중 "spec/06", "spec 조사" 등
- **영향도**: 중간. 20+5 = 25개 문장만 손보면 되므로 원본 텍스트 파일보다 범위는 작음.

### (3) 생성된 HTML 자체 — 재생성 방식 권장
- `기관/total/서울시_5개구청_사업제안기획안.html`은 (1),(2)를 고친 뒤
  `python3 build_html_report.py`로 재생성하는 것이 원칙(직접 HTML을 수정하면
  스크립트와 산출물이 어긋나 다음 재생성 때 되돌아감).
- 단, "지금 당장 HTML만 급히 고쳐야 한다"는 경우라면 HTML 내 텍스트만 정규식 치환하는
  임시 방편도 가능하나, 원본 텍스트 파일과의 동기화가 깨지므로 권장하지 않음.

## 4. 제안 실행 순서 (실행 시 참고용, 이번 요청에서는 미실행)

1. 정규식 치환 스크립트 작성:
   - `spec/(\d+)` → `조사자료\1` (또는 확정된 표기)
   - `plan/(\d+)` → `제안서\1`
   - `spec(?=[가-힣])` → `조사자료` (스페이스 없이 한글 조사가 바로 붙는 경우)
   - `plan(?=[가-힣])` → `제안서`
   - 단어 경계 처리에 주의: `bank_ideas_draft`, 영문 파일명, `data.nowon.go.kr` 같은
     실제 URL/파일명 안에 있는 spec/plan 유사 문자열은 치환 대상에서 제외해야 함
     (이번 조사에서는 URL 충돌 사례는 발견되지 않았으나 재확인 필요)
2. `기관/{구청}/spec/*.txt`, `plan/*.txt`, `bank_ideas_draft.txt` 80개 파일 일괄 치환
3. `build_html_report.py`의 `key_problems`/`verdicts` 딕셔너리 25개 문장 치환
4. `python3 build_html_report.py` 재실행으로 HTML 재생성
5. gstack-browse로 5개 구청 × 7개 서브탭 재검증(기존 검증 로그와 동일한 절차),
   특히 "조사자료07" 같은 신규 표기가 표/칩/불릿 등 좁은 레이아웃에서 줄바꿈 없이
   자연스럽게 보이는지 확인
6. `구청_log.md`에 치환 내역 및 재검증 결과 기록

## 5. 결정이 필요한 사항 (사용자 확인 필요)

- **번역어 최종 선택**: "조사자료" vs "조사보고서" vs "조사" / "제안서" vs "제안" vs "기획서"
- **파일 코드 표기 형식**: `조사자료07`(붙임) vs `조사자료 07`(공백) vs `조사자료#07`
- **원본 .txt 파일명 자체도 변경할지 여부**: 이번 계획은 파일 "내용"만 다루고
  `00_인덱스.txt` 같은 실제 파일명은 건드리지 않는 것을 전제로 함(파일명까지 바꾸면
  다른 문서에서 참조 중인 경로도 함께 깨질 수 있어 범위 확대 여부는 별도 확인 필요)
- **bank_ideas_draft.txt 같은 실제 파일명 인용**은 번역 제외 대상으로 둘지 확정 필요

## 6. 번호 ↔ 실제 파일명 매핑 (사용자 추가 요청 반영)

"번호도 실제 파일명을 지칭하면 좋겠다"는 요청에 따라, 단순히 `spec/07`→`조사자료07`처럼
숫자만 유지하는 게 아니라, **그 번호가 실제로 가리키는 파일명을 표기에 드러내는 안**을
아래와 같이 구체화한다.

### 6-1. `plan` 폴더 — 5개 구청 전부 번호 체계가 동일

| 번호 | 실제 파일명 | 짧은 한글 태그(안) |
|---|---|---|
| 00 | 제안개요_및_배경.txt | 제안개요 |
| 01 | 제안사업_요약표.txt | 제안요약표 |
| 02 | IT디지털기획_사업제안.txt | IT기획제안 |
| 03 | 금전적지원_사업제안.txt | 금전지원제안 |
| 04 | 실행로드맵_및_기대효과.txt | 실행로드맵 |
| 05 | 검증결과.txt | 검증결과 |

→ `plan/02`처럼 번호만 쓰던 표기를 `제안서-IT기획제안`, `제안서-검증결과`, `제안서-실행로드맵`
등으로 바꾸면 숫자를 몰라도 무엇을 가리키는지 문장만 보고 알 수 있다.
(5구청 공통이므로 이 매핑은 **단일 치환 규칙 하나**로 전체 적용 가능 — 실행 난이도 낮음)

### 6-2. `spec` 폴더 — 앞 2개·뒤 2개는 공통, 중간 사업목록 파일은 구청마다 다름

공통 부분(구청 무관):

| 번호 | 실제 파일명 | 짧은 한글 태그(안) |
|---|---|---|
| 00 | 인덱스.txt | 인덱스 |
| 01 | 4개년계획_개요.txt / 구정운영계획_개요.txt | 개요 |
| (마지막-1) | 홈페이지검색확인결과.txt | 검색확인결과 |
| (마지막) | 민원게시판_2026년정리.txt | 민원게시판 |

중간 부분(사업목록, 구청별로 상이 — **구청별 개별 매핑 필요**):

| 구청 | 02 | 03 | 04 | 05 | 06 | 07 |
|---|---|---|---|---|---|---|
| 도봉 | 1장 상생경제도시 | 2장 안전행복도시 | 3장 균형발전도시 | 4장 교육문화도시 | 5장 청렴행정도시 | (검색확인결과) |
| 노원 | 1장 미래도시 | 2장 교통도시 | 3장 문화도시 | 4장 힐링도시 | 5장 교육도시 | 6장 건강복지도시 |
| 광진 | 균형발전도시 | 상생복지도시 | 경제활력도시 | 문화교육도시 | 안전환경도시 | 열린소통도시 |
| 동대문 | 쾌적한동대문 | 즐거운동대문 | 민생동대문 | 안전한동대문 | 새로운동대문 | (검색확인결과) |
| 동작 | 세대별사업 | 복지도시분야 | 생활경제분야 | 교육역사문화분야 | 안전분야 | (검색확인결과) |

→ 이 부분은 spec/plan처럼 "한 가지 규칙"으로 전체를 못 바꾸고, **구청별로 별도 치환
매핑표를 하나씩 만들어야** 한다(노원구만 07까지 있고 나머지는 06 또는 07이 검색확인결과라
번호 밀림이 발생 — 치환 스크립트에서 구청 이름을 분기 조건으로 넣어야 함).
실행 시 리스크가 가장 큰 지점이므로, 실행 단계에서는 이 매핑표를 그대로 코드화해서
"번호+파일명"을 함께 다루는 조회 테이블(딕셔너리)로 만드는 방식을 권장.

### 6-3. 표기 형식(안) — 번호 대신 파일명을 앞세우는 방식

숫자를 아예 없애고 파일명 자체를 괄호로 병기하는 방식도 가능:

| 현재 | 안 A (숫자+한글 접두) | 안 B (파일명 병기, 번호 제거) |
|---|---|---|
| `spec/07` | `조사자료07` | `조사자료(검색확인결과)` |
| `spec/02(1장-4)` | `조사자료02(1장-4)` | `조사자료(사업목록·1장-4)` |
| `plan/05` | `제안서05` | `제안서(검증결과)` |
| `plan/02 IT-1 "..."` | `제안서02(IT기획제안) IT-1 "..."` | `제안서(IT기획제안) IT-1 "..."` |

**권장안**: 안 B(파일명 병기)가 "숫자를 몰라도 이해되는" 목적에 가장 부합하나, 문장 길이가
늘어나 표/칩처럼 좁은 레이아웃(`code-chip`, `kv-label`)에서 줄바꿈 문제가 생길 가능성이 있다.
이 경우 **본문 문단(`<p>`, `<li>`)에는 안 B**를, **좁은 표 칸·칩에는 안 A(숫자 유지)**를
쓰는 이원화 방식을 최종안 후보로 제안한다.

## 7. 'spec/plan' 외 다른 축약어 전수 검토 (사용자 추가 요청)

`build_html_report.py`가 만든 HTML에서 **HTML 마크업(태그명·속성명·class명·style/script
내부)을 모두 제거하고, 실제로 브라우저 화면에 문자열로 렌더링되는 텍스트 노드만**
`html.parser.HTMLParser`로 정확히 추출해 전수 확인했다(4,266개 텍스트 조각). 그 결과
영문 약어는 아래 세 그룹으로 명확히 나뉜다.

### 그룹 1 — 이 프로젝트가 만든 고유 코드 (spec/plan과 동일 성격, 번역 검토 대상)

| 표기 | 등장 횟수 | 의미 | 비고 |

**완전 한글화 시 오히려 코드로서의 간결성이 떨어질 수 있음** — 결정 필요 항목으로 별도 표기(§8).

### 그룹 2 — 실제 웹사이트 도메인/URL 경로/시스템 파라미터 (번역 대상 아님, 유지 필수)

제외 목록(deny-list)으로 별도 관리해야 함(6절 §4의 리스크 항목과 동일 맥락).
