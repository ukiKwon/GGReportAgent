IT/디지털 기획 사업 제안 (필수 포함 항목)

=================================================
[IT-1] 주요업무계획 상시공개 및 마스터DB 구축
=================================================
우선순위: 최우선
추정예산: 350~500백만원 (1회성 구축비 250~350 + 연 운영비 30~50 × 2년)
제안부서(추정): 기획예산과 + 정보통신팀 (구축), 전 부서(콘텐츠 제공)

[문제 정의]
spec/01 조사 결과, 동대문구는 도봉구식 "4개년계획" 단일 최상위 문서가 없고 매년 발간하는
"주요업무계획" 책자가 최상위 계획 역할을 함. 그런데 홈페이지 게시판에서 2024년, 2025년,
2026년판을 확인한 결과 실물 PDF가 게시되어 있지 않고 홍보 동영상(mp4)만 게시되어 있음이
확인되었고, 본 조사에서도 2023년판 PDF(80페이지) 외 최신 3개년 PDF를 발견하지 못함.
즉 주민이 "올해 구청이 무엇을 계획하고 있는지" 원문 문서로 확인할 방법이 3년 이상
사실상 막혀 있는 상태이며, 이는 검색 접근성 이전 단계의 "원본 문서 부재" 문제임.

[제안 내용]
1. 매년 발간되는 주요업무계획을 발간 즉시(늦어도 발간 후 30일 이내) PDF 원문을
   행정정보>시책안내 게시판에 의무 게시하도록 내부 지침화하고, 게시 여부를 정보통신팀이
   분기별로 점검하는 체크리스트 프로세스 수립(시스템이라기보다 운영 프로세스 정비가 핵심)
2. 2023년판 PDF(텍스트 레이어 정상 포함, 도봉구 문서와 달리 이미지 렌더링 불필요했던 점을
   활용)를 기준으로 122개 세부사업을 표준 메타데이터(사업명, 소관 장/전략, 담당부서, 기간,
   위치, 진행상태)로 정리한 마스터DB를 우선 구축하고, 이후 연도별 신규판이 게시될 때마다
   증분 갱신하는 구조로 설계
3. 민선9기(2026.7~) 5대 구정혁신방향(경제도시/교통도시/주거품격/복지도시/교육문화혁신도시,
   spec/01)이 정식 계획서로 확정되는 즉시 마스터DB의 상위 분류체계를 갱신할 수 있도록
   유연한 태그 구조(기존 5대 전략 태그 + 신규 혁신방향 태그 병행)로 설계
4. 홍보 동영상(mp4)은 유지하되, 동일 페이지에 원문 PDF·주요 내용 요약(텍스트)·사업별
   마스터DB 링크를 함께 제공하여 "영상만 있고 문서는 없는" 현재 상태를 해소

[기술 스택 권장]
- 게시 프로세스: 별도 신규 시스템보다 기존 게시판(www.ddm.go.kr) 첨부파일 기능 활용,
  정보통신팀 분기별 점검 체크리스트(저비용)
- 마스터DB: 경량 DB(PostgreSQL/MySQL) + 관리자 입력 UI, 열린구청장실·시책안내 페이지와 연동

[기대효과]
- "정책문서가 존재하지 않는다"는 구조적 신뢰 문제 해소, 향후 조사·감사·언론 질의 대응력 제고
- IT-2(통합 검색 플랫폼)의 데이터 소스로 즉시 활용 가능

=================================================
[IT-2] 동대문구 통합 사업정보·예산 검색 플랫폼
=================================================
우선순위: 최우선
추정예산: 400~550백만원 (구축비 300~400 + 연 운영비 50~75 × 2년)
제안부서(추정): 정보통신팀 + 기획예산과

[문제 정의]
spec/07 조사 결과, 122개 사업 중 표본 27개 검색 시 확인됨 52%, 부분확인 7%, 확인안됨 41%.
대형 개발사업·정례축제는 확인율이 매우 높은 반면(2장 즐거운동대문 83%), 생활�습착형
소규모 사업은 확인율이 33~40%대로 낮음(4장 안전한동대문 33%, 1장 쾌적동대문 40%).
근본 원인은 동대문구가 도봉구의 m.dobong.go.kr/majorWork 같은 "주요사업 추진현황" 전용
DB를 보유하지 않아 확인 경로가 전적으로 언론보도·서울시 포털에 의존한다는 점(spec/07 명시).
또한 spec/01에서 2023년판 PDF에는 사업별 연차별 투자예산표 자체가 없어, 개별 사업에
얼마의 예산이 배정되어 있는지 확인할 수 없는 정보공개 공백도 함께 존재함.

[제안 내용]
1. IT-1 마스터DB를 데이터 소스로 하여, PC/모바일 반응형 웹 기반 통합 검색 플랫폼 신설
   - 검색 UX: 사업명/키워드/5대 전략(쾌적·즐거운·민생·안전·새로운)/담당부서/진행상태 필터
   - 대형사업뿐 아니라 생활밀착형 사업(전기차충전 스마트관리, CCTV 인력확충 등 spec/07에서
     확인안됨으로 분류된 사업)도 동일한 비중으로 노출되도록 설계(현재 언론보도 의존 구조의
     역편향 해소)
2. 각 사업 페이지에 관련 예산 정보를 배정 가능한 범위 내에서 표기 - 사업 단위 예산표가
   원문에 없는 한계(spec/01)를 고려해, 최소한 소관 "분야별 예산"(따뜻한복지/쾌적한도시/
   친절한구정 등 8개 구분, 2023년 797,731백만원/2026년 982,436백만원) 수준까지는 연동
3. 서울시 관련 포털(정비사업 정보몽땅, 서울균형발전포털, 서울청년센터 등, spec/07에서
   확인 경로로 활용된 곳들)의 관련 페이지 링크를 사업별로 통합 노출해 "검색 경로 다변화"
4. 지방재정365/지방재정통계 등 외부 공개 재정지표(재정자립도 등 spec/01에서 확인안됨으로
   명시된 지표)를 API 또는 정기 수작업 갱신으로 함께 게시해 재정 투명성 보완

[기술 스택 권장]
- 검색엔진: 경량 오픈소스(Meilisearch 등)
- 프론트엔드: 반응형 웹, SSR 적용으로 검색엔진 색인 대응
- 백엔드: IT-1 마스터DB 재활용, 신규 개발 최소화

[기대효과]
- 122개 사업 확인율 52% -> 목표 80% 이상으로 개선(특히 생활밀착형 사업 확인율 개선 목표)
- 재정자립도 등 미확인 지표를 상시 공개해 정보공개 완전성 제고

[IT-3] 민원 채널 통합 대시보드 & 처리현황 알림 서비스
=================================================
우선순위: 최우선
추정예산: 300~450백만원 (구축비 250~350 + 연 알림 발송비 50~100)
제안부서(추정): 감사담당관(옴부즈만 소관) + 정보통신팀

[문제 정의]
spec/08 조사 결과:
1. 실사용 민원 채널이 "구청장에게 바란다"(chief.ddm.go.kr), "옴부즈만 민원신청"
   (www.ddm.go.kr), "주민참여예산제도" 등으로 분산되어 있고, 옴부즈만 접수 민원 상당수가
   "구청장에게 바란다"로 이첩되는 구조이나(2026년 이첩완료 다건 확인) 이첩 후 최종 처리
   결과까지는 옴부즈만 게시판에서 추적되지 않음
2. "구청장에게 바란다" 조회 시점(2026.7.16) 상위 10건 전부 "처리중" 상태로 표시되고
   "답변완료" 표시가 목록에 노출되지 않아 주민이 처리경과를 추적하기 어려움
3. "공직자부조리신고"·"하도급부조리신고" 게시판은 정상 작동하나 게시글이 0건으로,
   실제 신고 활성도가 낮거나 별도 채널(국민권익위 청렴포털 등)로 신고가 분산되고 있을
   가능성이 있으나 이를 파악할 통합 통계 체계가 없음

[제안 내용]
1. "구청장에게 바란다", "옴부즈만 민원신청", "공직자·하도급 부조리신고" 3개 채널의
   접수번호를 통합 관리하는 백엔드 대시보드 구축 - 이첩 처리(옴부즈만→구청장에게 바란다)
   이력을 단일 화면에서 추적 가능하도록 설계
2. 주민용 "내 민원 조회" 페이지: 접수번호+휴대폰 인증으로 채널·이첩 여부와 무관하게
   진행상황 실시간 확인(접수 -> 담당부서 배정 -> 검토중 -> 답변완료 4단계 공통 상태값)
3. 카카오 알림톡 연동: 상태 변경 시(특히 답변완료) 자동 알림 발송 - "구청장에게 바란다"처럼
   목록에 처리결과가 안 보이는 채널에 우선 적용
4. 부조리신고 게시판(현재 0건)의 활성도 저조 원인을 파악하기 위해, 연 1회 익명 신고건수
   통계(청렴포털 등 타 채널 이첩분 포함 추정치)를 공직자 청렴교육 자료로 활용
5. 연 1회 "민원처리 통계 공표"를 옴부즈만 자료실 형태로 자동 생성 - 채널별 평균 처리일수,
   유형별 건수 등 투명 공개

[기술 스택 권장]
- 기존 시스템(chief.ddm.go.kr, www.ddm.go.kr selectCvplRceptWebList 계열) API/DB 연동
  어댑터 개발(전면 교체보다 연동 방식이 저비용)
- 알림톡: 카카오 비즈메시지 API

[기대효과]
- 민원 처리 투명성 확보, 이첩 후 추적 공백 해소
- 2026년 최다 민원(신답극동아파트 관련)의 처리상황을 주민이 직접 확인 가능해져 반복 민원 감소

[IT-4] 노후공동주택 안전이슈 조기경보 시스템
=================================================
우선순위: 상
추정예산: 150~220백만원
제안부서(추정): 건축안전센터 + 정보통신팀

[문제 정의]
spec/08: 조사일(2026.7.16) 기준 "구청장에게 바란다" 최신 10건 중 8건이 "신답극동아파트"
리모델링조합 관련 안전진단·행정명령·소방차 통행확보 요청 등에 집중됨. 이는 특정 단지의
안전 관련 갈등이 단기간에 다수 민원인의 반복 제기로 집중 폭증한 사례로, spec/02(1-7 소규모
노후건축물 안전점검, 1-6 공동주택 관리지원)에는 이런 개별 단지 갈등에 대한 조기 감지·
신속 대응 프로세스가 명시되어 있지 않음.

[제안 내용]
1. IT-3 민원 대시보드와 연동해, 특정 단지·주소 기준으로 민원이 일정 건수(예: 1주일 내 5건
   이상) 집중될 경우 건축안전센터·주택과에 자동 경보(대시보드 하이라이트 + 알림)하는
   조기경보 로직 구축
2. 경보 발생 시 건축안전센터의 기존 소규모 노후건축물 합동점검(spec/02 1-7) 절차를
   우선순위 조정하여 해당 단지에 조기 투입하는 대응 매뉴얼 수립
3. 신답극동아파트 사례처럼 리모델링조합-입주민 간 갈등이 있는 단지는 별도 태그로 분류해
   법률·건축 전문가 자문(spec/02 1-1 재개발·재건축 전문가 인력풀 18명 활용) 연계 경로 마련

[기대효과]
- 특정 단지發 민원 집중 상황을 조기에 포착해 대규모 갈등으로 확산되기 전 행정 개입 가능
- 기존 전문가 인력풀(spec/02 1-1)을 재활용해 별도 인력 충원 없이 대응력 확보
