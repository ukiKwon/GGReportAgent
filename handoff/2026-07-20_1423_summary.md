## 새 Agent 설계 방향 (사용자 확정)

- **입력**: `GGReportAgent/RFP/` 안의 새 RFP(공고문) PDF.
- **스타일 참고 코퍼스**: 5개 구청 spec/plan/bank_ideas 자료. 단, "새 기관의 실제 데이터"가
  아니라 **문서 포맷/톤 템플릿**으로만 사용(예: 번호 매김 섹션, "근거자료: spec/NN" 인용
  방식, 축(A/B/C) 구조, 요약표, Top3+비고 블록 패턴).
- **신규 기관 조사**: 새 RFP의 대상 기관(예: 수원시)은 기존 5개 구청과 다른 기관이므로, 그
  기관에 대한 spec 상당 자료는 agent가 **웹검색(WebSearch/WebFetch)으로 직접 수집**해야 함.
  proj2에는 이 노드가 없으므로 새로 설계 필요.
- **배점표 처리**: RFP의 배점표(평가 항목별 배점)를 추출해 제안서의 章 구성/레이아웃을 그
  항목에 맞춰 구성하고, 배점표 자체도 산출물에 포함. PDF 텍스트 추출은 pypdf 우선, 실패 시
  PyMuPDF 이미지 렌더링 + Vision LLM fallback(도봉구 PDF가 CID 폰트라 이미지 렌더링이
  필요했던 선례가 있음 — 구청_log.md 14:11~14:29 참고).
- **산출물 포맷**: PPTX 슬라이드(문서형 docx/html이 아님) — proj2의 pptx_builder.render_pptx
  패턴 재사용 방향.
- **위치**: `agent/` 아래 새 패키지로 구축, 생성 산출물은 `report_new/`에 저장.
- **개발 방식**: superpowers 표준 흐름 — brainstorming → spec 문서 저장 → writing-plans →
  TDD.
