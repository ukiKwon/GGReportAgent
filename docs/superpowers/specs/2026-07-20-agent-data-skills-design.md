# Agent 데이터 Skill 설계: rfp-locate + institution-corpus-format

## Context

`GGReportAgent`는 **지방자치단체 및 공공기관**을 대상으로 KB 파트너십 제안서를 만드는
프로젝트다. 지금까지는 서울시 5개 구청(도봉/노원/광진/동대문/동작)을 대상으로 사람이
대화형 세션 + 백그라운드 리서치 에이전트로 수작업 제작한 결과물이 쌓여 있다
(`{district}/spec/`, `{district}/plan/`, `{district}/bank_ideas_draft.txt`, `total/*.docx|html`).
이 5개 구청 자료는 앞으로 만들 agent가 참고할 **완성된 포맷 템플릿**이지, agent의 대상
범위를 구청으로 한정하는 것이 아니다 — agent는 구청뿐 아니라 시/도, 공기업, 공공기관 등
RFP(공고문)를 내는 모든 지자체·공공기관을 대상으로 동작해야 한다.

이전 세션에서 사용자와 함께 "새 RFP가 주어지면 과거 작업 스타일을 참고해 유사한 제안서를
자동 생성하는 agent"의 큰 방향(입력=RFP PDF, 참고=5개 구청 자료를 포맷 템플릿으로 사용,
신규 기관 조사=웹검색, 배점표 기반 章 구성, 출력=PPTX, 위치=`agent/`, 산출물=`report_new/`)을
확정했다(`handoff/2026-07-20_1423_summary.md` 참고).

이번 세션은 그 agent 전체가 아니라, **agent가 사용할 데이터를 다루는 두 개의 독립된 skill**을
정의하는 것이 목적이다:

1. RFP PDF를 찾고 텍스트/배점표를 추출하는 skill
2. 지자체·공공기관 공통으로 쓰는 spec/plan/bank_ideas_draft 산출물의 표준 포맷을 정의하는
   skill(기존 5개 구청 포함 일반화 — 특정 기관 전용이 아니라 포맷 규칙 자체를 규정)

두 skill 모두 `agent/` LangGraph 노드가 아니라, Claude Code의 `Skill` 메커니즘(`SKILL.md` +
번들 스크립트/레퍼런스)으로 구현되어, agent 구현 여부와 무관하게 대화형 세션에서도 바로
쓸 수 있어야 한다.

## Skill 1: `rfp-locate`

### 목적
RFP(공고문) PDF를 찾고, 본문 텍스트와 배점표(평가 항목별 배점)를 추출해 구조화된 JSON으로
남긴다. 후속 분석(요구사항 추출, 章 구성 설계)이 이 산출물을 입력으로 삼는다.

### 트리거
- "RFP 찾아줘", "배점표 추출해줘"
- 새 지자체·공공기관 제안서 작업을 시작할 때 자동 진입

### 흐름
1. **탐색**: `GGReportAgent/RFP/` 폴더를 스캔한다.
   - PDF가 정확히 1개면 그것을 사용한다.
   - 여러 개면 사용자에게 어떤 파일인지 확인을 요청한다(추측 금지).
   - 폴더가 비어있으면 2번으로.
2. **웹서치 폴백** (폴더가 비어있을 때만):
   - 사용자가 제공한 기관명/공고명으로 원본 공고 PDF를 웹서치로 탐색한다.
   - 찾으면 다운로드해 `RFP/`에 저장하고 사용자에게 확인받는다.
   - 못 찾으면 **절대 추측하지 말고** 사용자에게 파일을 직접 업로드하거나 경로를 알려달라고
     요청한다.
3. **텍스트 추출**: 번들 스크립트 `scripts/extract_text.py`로 pypdf 기반 텍스트 추출을 우선
   시도한다.
4. **실패 판정 및 폴백**: 추출된 텍스트가 다음 중 하나에 해당하면 비정상으로 판단한다:
   - 텍스트 길이가 페이지 수 대비 비정상적으로 짧음(예: 페이지당 평균 50자 미만)
   - 깨진 문자(CID 참조, 유니코드 대체문자 `�`)가 다수 포함됨
   - 이 경우 `scripts/render_pages.py`(PyMuPDF)로 페이지를 이미지로 렌더링하고, Claude가
     Read 도구(Vision)로 직접 읽어 텍스트/배점표를 해석한다. (도봉구 RFP가 CID 폰트라
     이미지 렌더링이 실제로 필요했던 선례가 `구청_log.md`에 있음.)
5. **배점표 구조화**: 텍스트(또는 vision 판독) 결과에서 평가 항목/배점/설명을 아래 JSON
   스키마로 추출한다:
   ```json
   {
     "institution": "수원시",
     "rfp_title": "수원시 금고 지정 계획 공고문",
     "total_score": 100,
     "criteria": [
       {"category": "재무 건전성", "item": "자기자본비율", "score": 10, "description": "..."}
     ]
   }
   ```
6. **산출물 저장**:
   - `report_new/{institution}/rfp_scoring.json` — 위 스키마
   - `report_new/{institution}/rfp_text.txt` — 추출된 원문 텍스트(또는 vision 판독 텍스트)

### 포함 파일
- `SKILL.md` — 위 절차를 단계별 지시문으로 작성
- `scripts/extract_text.py` — pypdf 텍스트 추출 + 비정상 판정 휴리스틱
- `scripts/render_pages.py` — PyMuPDF 페이지 이미지 렌더링
- `references/scoring_schema.json` — 배점표 JSON 스키마 예시 1개(수원시 사례 활용 가능)

### 에러 처리 원칙
- 파일을 못 찾거나 텍스트 추출이 애매할 때 **절대 추측하지 않고** 사용자에게 확인을 요청한다.
- 배점표가 RFP에 아예 없는 공고문일 수도 있음 — 이 경우 `criteria: []`로 남기고 사용자에게
  배점표 없음을 보고한다(빈 배열을 실패로 취급하지 않는다).

## Skill 2: `institution-corpus-format`

### 목적
지자체·공공기관(신규 기관 + 기존 5개 구청 모두)에 대한 `spec/`, `plan/`,
`bank_ideas_draft.txt`를 작성/검토할 때 따라야 할 표준 문서 포맷과 인용 규칙을 정의한다.
이 skill은 **포맷 규칙**만 규정하며, 실제 데이터를 채우는 조사/집필 자체는 별도 작업
(웹서치 agent 또는 사람)이 수행한다.

### 트리거
- 새 지자체·공공기관의 조사 결과를 spec 파일로 정리할 때
- 제안서(plan) 문서를 작성할 때
- 은행 결합 아이디어를 draft로 정리할 때
- 기존 5개 구청 자료를 참고 템플릿으로 재확인할 때

### 포맷 규칙

**`{institution}/spec/`** (원자료, 순번 파일 — 순번은 고정, 장 이름은 기관마다 다를 수 있음):

| 파일 | 내용 |
|---|---|
| `00_인덱스.txt` | 각 파일 한 줄 요약 + 전체 개수/예산 합계 자체 검산(장별 합 = 총계 형태로 명시) |
| `01_개요.txt` | 기관 계획 개요, 비전/목표, 장 구조, 총 예산 |
| `0N_{장명}_사업목록_예산.txt` | 사업별 번호(`N-M`)·예산·연차별 표. 장 이름/개수는 기관 고유 계획 구조를 따른다 |
| `0N_홈페이지검색확인결과.txt` | 각 사업을 웹서치로 대조해 **확인됨/부분확인/확인안됨** 3단계로만 분류. "확인안됨"은 폐기·보류를 의미하지 않고 검색 한계일 수 있음을 명시 |
| `0N_민원게시판_YYYY년정리.txt` | 최신 연도 민원 채널별 현황과 유형 패턴 |

신규 지자체·공공기관은 이 spec 원자료를 **웹서치로 직접 수집**해서 채운다. 기존 5개 구청
(도봉/노원/광진/동대문/동작) 자료는 이미 완성된 참고 템플릿으로 취급하고 재수집하지 않는다.

**`{institution}/plan/`** (제안 문서 — 모든 주장은 spec을 인용):

| 파일 | 내용 |
|---|---|
| `00_제안개요_및_배경.txt` | 제안 취지, 배경 |
| `01_제안사업_요약표.txt` | 표 컬럼: 번호(`IT-N`/`FN-N`)/사업명/우선순위/추정예산/담당(추정)/**연계 근거(spec/NN)** |
| `02_..._사업제안.txt`, `03_..._사업제안.txt` 등 | 카테고리별 상세 제안. 카테고리는 RFP의 배점표 항목(Skill 1의 `rfp_scoring.json`)에 맞춰 조정 가능 — 도봉 사례의 "IT디지털기획/금전적지원" 2분류는 고정 스키마가 아니라 예시임 |
| `04_실행로드맵_및_기대효과.txt` | 실행 일정, 기대효과 |
| `05_검증결과.txt` | 사업별 **중복여부/타당성등급/판단근거/수정권고사항** + 전체 신뢰도 점수(/100). 판단근거는 spec 원문과 대조해 인용 정확성까지 검증하고, 틀린 인용은 반드시 지적해 수정권고사항에 반영 |

**`{institution}/bank_ideas_draft.txt`**:
- 상단에 "초안(draft) — 정식 반영 전, plan 반영 여부 별도 검토 필요" 명시
- "근거자료: {institution}/spec/00~NN, {institution}/plan/00~NN" 형태로 참조 범위 선언
- 축 구조로 분류(소상공인금융/청년금융/IT협력/복지주택금융/SOC참여 — 기관 특성에 따라 축 자체는
  조정 가능하나, 각 아이디어는 반드시 축에 속해야 함)
- 아이디어별 필수 3블록: `연계 구청사업/근거`(spec/plan 인용) → `구체적 상품/협력 형태` →
  `은행 기대효과`
- 상품명은 실제 금융상품 브랜드명이 아닌 일반 카테고리(정책연계대출, 매칭적금 등)로만 표기

### 관통 규칙 (인용)
모든 수치·사실 주장은 `spec/NN` 또는 `spec/NN(장-M)` 또는 `plan XX-N` 형태로 출처를 명시해야
한다. 검증 문서(`05_검증결과.txt`)에서 이 인용이 spec 원문과 실제로 일치하는지 대조하며,
불일치·과장·오인용을 발견하면 등급을 낮추고 수정권고사항에 명시한다(도봉 IT-3 사례:
"확인됨"으로 인용했으나 spec 원문은 "부분확인"이었던 오류를 검증 단계에서 지적).

### 포함 파일
- `SKILL.md` — 위 포맷 규칙을 파일 생성 체크리스트 + 인용 규칙으로 작성
- `references/dobong_spec_sample.txt`, `references/dobong_plan_sample.txt`,
  `references/dobong_bank_ideas_sample.txt` — 기존 도봉구 파일에서 발췌한 실제 포맷 예시
  (신규 기관 작업 시 형식 참고용, 도봉구 데이터 자체를 복사해 쓰지 않도록 skill 안에서
  "예시일 뿐 원본 그대로 재사용 금지" 명시)

## 두 Skill의 관계

- `rfp-locate`는 **입력**(새 기관의 RFP)을 다루고, `institution-corpus-format`은 **출력**(그
  기관에 대한 spec/plan/bank_ideas_draft)을 다룬다 — 서로 독립적으로 호출 가능하고, agent
  파이프라인에서는 순서대로 사용된다(RFP 배점표 → plan의 章 구성 카테고리 결정에 참고).
- 두 skill 모두 `agent/` LangGraph 구현과 무관하게 대화형 세션에서 단독으로도 유용해야 한다
  (예: 사람이 직접 새 구청 자료를 조사하며 이 skill들을 참고해 수작업으로 spec/plan을 쓸 수도
  있음).

## 테스트/검증 계획

- 코드 산출물이 아닌 skill(`SKILL.md` + 레퍼런스 + 스크립트)이므로 별도 자동 테스트 스위트는
  없다.
- `rfp-locate`의 번들 스크립트(`extract_text.py`, `render_pages.py`)는 순수 유틸리티이므로
  `RFP/수원시 금고 지정 계획 공고문.pdf`를 대상으로 실제 실행해 텍스트 추출 결과를 눈으로
  확인하는 수동 검증을 수행한다(TDD 대상은 이 두 스크립트로 한정).
- `writing-skills` 스킬의 자체 검증 체크리스트(플레이스홀더 없음, 실행 가능한 지시문인지)를
  두 skill 모두에 적용한다.
