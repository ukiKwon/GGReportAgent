# 5-report 대시보드 설계

작성일: 2026-07-18
대상 파일: `report/5-report.html` (기존에 빈 파일로 존재하던 것을 채움)
소스 데이터: `report/5-report.md`, `{구}/bank_idea_draft.txt` (도봉·노원·광진·동대문·동작 5개구)

## 1. 목적 및 용도

`report/5-report.md`(서울시 5개 구청 사업제안 기획안)의 내용을 은행/구청 담당자 등 **외부 발표 자리에서 한눈에 비교해서 보여줄 수 있는 대시보드**를 만든다. 브레인스토밍 과정에서 결정된 핵심 방향은 다음과 같다.

- **핵심 메시지**: 5개 구를 나란히 비교하는 것이 최우선 — 어느 구가 가장 탄탄한지/리스크가 큰지 즉시 보여야 함
- **드릴다운**: 비교 화면에서 구를 클릭하면 대시보드 안에서 상세 내용(IT/금전지원 사업 7건, 검증표, 은행 결합 아이디어 전체 목록)까지 볼 수 있어야 함 — 원본 md를 별도로 참고하게 하지 않음
- **전달 형태**: 로컬 HTML 파일 하나. 인터넷 연결 없이도 완전히 작동해야 하고(오프라인 발표 대비), 파일 하나만 옮기면 어디서든 열림

## 2. 확정된 화면 구조

브라우저 목업으로 검증한 구조:

```
┌─────────────┬──────────────────────────────────────────┐
│ 🗺 대시보드  │                                            │
│ (active)    │         [메인 영역]                        │
│             │                                            │
│ 📊 리포트   │                                            │
└─────────────┴──────────────────────────────────────────┘
```

### 2-1. 대시보드 탭 — 서울 25개 자치구 지도

- 서울시 **25개 자치구 전체**를 실제 상대적 위치(강북/강남, 인접관계)에 맞춰 SVG로 배치
- 이번 프로젝트 대상인 5개 구(도봉·노원·광진·동대문·동작)만 **색이 있고 데이터가 채워진 "활성(active)" 상태**, 나머지 20개 구는 **회색 "빈 자리(placeholder)"** — 클릭하면 "아직 대상 구가 아닙니다" 안내만 뜨고 자리와 클릭 동작은 이미 준비되어 있음
- 활성 구를 클릭하면 **화면 전체를 덮는 오버레이**로 확장(페이지 이동 없이 애니메이션으로 펼쳐짐)

### 2-2. 구 상세 화면 (전체화면 오버레이)

브레인스토밍 중 정한 대로, 세로로 길게 이어지는 한 페이지 대신 **내부에 서브탭 3개**로 나눈다(스크롤 부담 완화):

1. **핵심요약** — spec 조사에서 확인된 핵심 문제, 신뢰도 점수, 검증 총평
2. **IT·금전사업** — IT/디지털 사업 4건 + 금전지원 사업 3건 카드(사업명·예산·우선순위·근거), 검증표(중복여부·타당성등급·판단근거·수정권고사항)
3. **은행아이디어** — bank_idea_draft.txt의 전체 아이디어 목록을 카테고리별(소상공인금융/청년금융/IT협력/주택금융/PF 등)로 묶어서 표시, Top3 추천은 강조 표시

"← 지도로 돌아가기" 버튼으로 대시보드 탭으로 복귀.

### 2-3. 리포트 탭 — 주제별 비교

- 상단 드롭다운에서 주제 선택: `IT사업 / 금전지원 사업 / 검증 신뢰도 점수 / 은행 결합 아이디어`
- 선택한 주제를 기준으로 **활성 구들만** 그리드로 나란히 비교 표시(placeholder 구는 데이터가 없으므로 비교 대상에서 자동 제외)
- 각 열에 무엇을 보여줄지(모호성 해소):
  - `IT사업`/`금전지원 사업`: 해당 구의 항목 전부(IT는 4건, 금전지원은 3건)를 사업명·예산·우선순위 압축 리스트로 세로로 쌓아서 표시 — 최우선 1건만이 아니라 전체를 비교
  - `검증 신뢰도 점수`: 점수 숫자 + 검증 총평 한 줄
  - `은행 결합 아이디어`: 카테고리 수 + Top3 제목만 요약 표시(전체 목록은 지도 쪽 구 상세화면에서 확인 — 리포트 탭은 "비교"가 목적이라 전체 목록을 5열에 다 펼치면 오히려 한눈에 안 들어옴)

## 3. 확장성 — 20개 구 추가를 감안한 데이터 설계

**오늘 결정된 사항**: 지금 당장 나머지 20개 구를 만들지는 않지만, 이번 프로젝트와 같은 폴더 구조(`{구}/plan/`, `{구}/spec/`, `{구}/bank_idea_draft.txt`)가 다른 구에도 나중에 추가될 수 있음을 전제로 설계한다. 이를 위해 지도 렌더링 로직과 데이터를 분리한다.

### 3-1. 데이터 스키마

모든 구(활성 5개 + 대기 20개)를 하나의 객체에 동일한 키 구조로 등록한다:

```js
const DISTRICTS = {
  dobong: {
    name: "도봉구",
    active: true,                 // false면 지도에서 회색, 클릭 시 안내만
    mapCell: "dobong",            // SVG 상의 자치구 자리와 매칭되는 id
    score: 74,
    budget: "20~30억원 (구 예산 대비 0.25~0.4%)",
    issues: ["...", "...", "..."],          // 핵심요약 탭
    itProjects: [ {code, name, budget, priority, basis}, ... ],   // IT·금전사업 탭
    fnProjects: [ {code, name, budget, priority, basis}, ... ],
    validation: [ {code, overlap, grade, rationale, recommendation}, ... ],
    verdictScore: 74,
    verdictBasis: "...",
    bankIdeas: {
      categories: { "소상공인금융": [...], "청년금융": [...], ... },
      top3: [...]
    }
  },
  // ... 노원/광진/동대문/동작 동일 구조

  eunpyeong: { name: "은평구", active: false, mapCell: "eunpyeong" },
  gangbuk:   { name: "강북구", active: false, mapCell: "gangbuk" },
  // ... 나머지 18개 구도 이름과 mapCell만 등록(빈 placeholder)
}
```

- **지금 채워야 할 것**: 5개 활성 구의 전체 필드(5-report.md와 bank_idea_draft.txt에서 직접 옮겨 적음)
- **나머지 20개 구**: `name`과 `mapCell`만 있으면 지도에 회색 자리로 나타나기에 충분

### 3-2. 나중에 새 구를 추가하는 절차 (문서화만, 지금 실행 안 함)

1. `{새구}/plan/`, `{새구}/spec/`, `{새구}/bank_idea_draft.txt`가 생기면
2. `DISTRICTS` 객체에서 해당 구의 `active`를 `true`로 바꾸고 위 스키마의 나머지 필드를 채움
3. 지도 SVG나 렌더링 함수, 리포트 탭 로직은 **손댈 필요 없음** — 이미 25칸이 다 그려져 있고, 렌더링 코드는 `active` 플래그만 보고 색칠·클릭 가능 여부를 결정하기 때문

이렇게 하면 지금은 A안(수동으로 데이터 옮겨 적기)으로 충분하고, 나중에 구가 늘어나도 매번 지도를 다시 그리거나 코드를 고칠 필요 없이 데이터만 추가하면 된다.

## 4. SVG 지도

- 25개 자치구 자리를 브레인스토밍에서 확정한 배치(강북 14개/강남 11개, 한강 기준 위아래)로 미리 다 그려둠
- 각 자치구는 `class="gu active"` 또는 `class="gu placeholder"`로 구분되고, 렌더링 시 `DISTRICTS` 데이터의 `active` 값과 동기화
- 참고: 이번 브레인스토밍 목업은 좌표를 손으로 대략 맞춘 것이며, 실제 파일에서도 동일한 손그림 좌표를 사용한다(정밀 행정경계 좌표 데이터를 별도로 구하지 않음 — 발표용 목적에 지금 정확도로 충분하다고 판단, 필요 시 추후 정밀화 가능)

## 5. 스타일

- 다크/라이트 모드 모두 지원(`prefers-color-scheme` 기준, 목업에서 쓴 다크 톤을 기본값으로)
- 외부 CDN·폰트·라이브러리 의존 없이 완전히 자립형(파일 하나로 오프라인 작동)
- 창 너비를 줄여도 레이아웃이 깨지지 않도록 최소한의 반응형 처리

## 6. 데이터 흐름

- 런타임 API 호출이나 서버 없음. 모든 데이터는 파일 상단 `<script>`에 `DISTRICTS` 객체로 내장
- 클릭 등 사용자 상호작용은 순수 JS로 DOM을 토글하는 방식(페이지 이동 없음)

## 7. 예외 처리

- 정적 파일이라 네트워크 에러 등은 해당 없음
- 비활성(회색) 구 클릭 시 "아직 대상 구가 아닙니다" 안내만 표시(이미 목업에서 확인)
